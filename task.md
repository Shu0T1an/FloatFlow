# FloatFlow Task Board

## 1. 背景与目标

当前 FloatFlow 已从“大面板工具”重构为右下角桌边挂件。
本阶段目标不是继续扩功能，而是把现有挂件整理成一个可验收、可继续迭代的稳定版本。

本轮稳定化重点：
- 修正透明度语义
- 增强 `Pin` 与右下角吸附反馈
- 补齐设置页信息
- 清理用户可见中文乱码
- 收口验证链路

## 2. 当前范围

本阶段已纳入：
- 右下角桌边挂件布局
- `Todo / 便笺` 双模式
- `Pin`
- 挂件视觉透明度
- `Alt + Space` 呼出/隐藏
- `Alt + N` 快速进入主录入入口
- JSON 持久化
- 图示风格独立设置窗口与基础说明

本阶段明确不做：
- 拖动
- 位置记忆
- 自由吸附
- 便笺删除
- 便笺归档
- 更复杂的桌面模式

## 5. 风险与限制

- 当前还没有完成真实桌面环境下的手动验收，现阶段结论主要来自测试、构建和 `cargo check`
- 当前版本位置固定，不支持拖动、位置记忆或自由吸附
- 当前独立设置窗口首版只承接已有设置能力与只读信息，未扩展到开机启动、语言、多端同步等完整设置中心能力

## 10. 2026-04-08 设置窗口白屏 Issue 结论
- [x] 根因确认：Windows 下的 Tauri 运行时窗口如果在同步 `#[tauri::command]` 中创建，容易在 `WebviewWindowBuilder::build()` 阶段卡住；本次症状正是 `open_settings_window` 停在创建链路，导致设置窗口白屏且后续聚焦 / 关闭流程无法继续。
- [x] 已修复：将 `open_settings_window` 与 `close_settings_window` 改为 `async` command，并保留现有 debug log 继续观察窗口创建、聚焦和关闭链路。
- [x] 已补回归保护：`src-tauri/src/lib.rs` 新增 Rust 测试，约束窗口管理命令保持 `async`，避免后续重构时把这个死锁问题重新引回。
- [ ] 待真实环境复验：重新执行 `npm run tauri dev`，点击 `系统设置` 后确认日志继续出现 `ensure_settings_window:created` / `open_settings_window:success`，并验证窗口不再白屏、标题栏和页面返回按钮都可以正常关闭。
